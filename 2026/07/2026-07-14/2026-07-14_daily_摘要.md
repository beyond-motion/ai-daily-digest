## 🧠 antirez："控制想法，而不是代码"——Redis 之父对 AI 编程的彻底摊牌

**来源**：antirez.com · 2026-07-13 11:39

Redis 创始人 Salvatore Sanfilippo（antirez）今天发了一篇在 Hacker News 上冲到 14 万+ 浏览的长文，标题只有一句话：**"Control the ideas, not the code."** 这是他对"AI 时代的程序员到底该干什么"这个话题迄今为止最完整、最不留退路的一次表态。

### 为什么要说"不中听"的话

antirez 首先解释了自己为什么反复谈论 AI 编程——不是为了"刷存在感"：

> "我是一名还算有声望的程序员，我不需要靠这些来证明自己。我重新回到了 Redis，也在开发新的本地 LLM 推理开源项目（DwarfStar）。我之所以一直说这些不讨喜的话，是因为我感到有必要帮助那些对变化准备不足的人——通常比我年轻——去降低这场冲击的影响。"

他的策略是一个"花招"（他自己用的词）：作为一个有威望的资深程序员站出来说"看，我会写代码，我不是躲在 AI 后面——但时代变了"。这样年轻开发者就不会觉得自己在"背叛"这个领域。

### 三个核心论点

1. **代码量爆炸，逐行 review 已不现实。** "你每天要 review 5000 行代码？怎么做？"

2. **LLM 擅长局部最优，人类应该关注全局设计。** "与其逐函数逐行扫描，不如去 prompt 你脑海中的设计，问'这部分的正确设计是什么？它怎么工作？'，然后评估模型是否正确。这快得多。"

3. **工作时间是 8 小时，看代码是零和博弈。** "如果你在读代码，你就少了时间去想更重要的事：这个软件到底在做什么？我要往哪个方向走？还有什么新的优化思路？还有大量的 QA 要做。"

### Mythical Man Month 的回响

antirez 引用了 1975 年 Fred Brooks 的经典：

> "一本 70 年代的书，比 2000-2020 年间说过的很多话都更准确地描述了当前的软件时代。"

他反问：那些现在抗议 AI 的人，为什么在过去十年软件质量已经崩坏的时候没有感到恐惧？

### 实践验证：DwarfStar 项目

antirez 用自动化方式实现了 DeepSeek v4 和 GLM 5.2 的推理引擎，但他强调这不是"说一句 implement XYZ 就完事"——你必须理解设计、评估性能、对比正确性。他发现现有推理实现中充斥着"累积损害"——索引 attention 实现做多了不必要的工作，导致上下文超过某个阈值后性能持续下降。

### 对 Redis 自己代码的态度

> "是的，我会发现不满意的地方并修改。但我打开其他 Redis 贡献者写的文件，问题更多——不是因为他们不是好程序员，而是品味问题……我现在做这件事（review），是出于对用户的尊重。但如果我完全自由，我会把所有 review 的时间用来做更多 QA、思考下一个优化点并实现它。"

**一句话总结：antirez 认为，在这个时代，程序员的产出不应该再以"你写了多少代码"或"你读了多少代码"来衡量——而是以"你把控了多少想法"来衡量。**

## 🎮 DOOMQL：用 SQLite 当游戏引擎，GPT-5.6 Sol 写出来的"不合理"项目

**来源**：Simon Willison's Weblog · 2026-07-13 22:34

Peter Gostev 用 GPT-5.6 Sol 构建了 DOOMQL——一个**用 SQLite 作为完整游戏引擎**的 Doom-like 游戏。不是"用 SQLite 存数据"，而是用 SQL 实现移动、碰撞、敌人、战斗、进度系统和每一帧的 RGB 像素渲染。

### 技术实现

整个游戏是一个 Python 终端脚本，核心是一个**用递归 CTE 实现的 SQLite 光线追踪器**。运行后会在 `/tmp/doomql/.doomql/doomql.sqlite` 创建数据库。

Willison 亲自试了一下，还用 Datasette 的全新 **Datasette Apps** 插件（支持在 Datasette 界面内运行自定义 HTML+JS 应用）配合 Claude Fable 5 做了一个实时画面渲染器：

> "我创建了一个新 app，把 copy-paste prompt 丢进 Claude chat（Fable 5），说：Build an app that displays the current state of the screen using the frame_pixels view with its x, y, r, g, b columns. have it refresh once a second."

然后加了一句 "add a minimap"，就得到了一个在 Datasette 里实时反映终端游戏画面的可视化界面。

### 为什么值得注意

这不只是一个"酷项目"——它是**当前 AI 辅助编程能力的一个极端测试用例**。用 GPT-5.6 Sol 生成一个完整的、用 SQL 实现光线追踪的游戏引擎，然后跨工具链（GPT-5.6 Sol → Python 终端 → SQLite → Datasette Apps → Claude Fable 5 做前端）串联出一个可玩的系统。这恰好是 antirez 说的"控制想法，不控制代码"的一个活体演示。

🔗 https://simonwillison.net/2026/Jul/13/doomql/ · https://github.com/petergpt/doomql

## 🔒 CISA GitHub 泄密复盘：美国网安机构忽视了 9 条自动告警，泄露 6 个月

**来源**：Krebs on Security · 2026-07-13 15:03

美国网络安全和基础设施安全局（CISA）发布了一份事故复盘报告，关于一个承包商在**公开 GitHub 仓库**中暴露了 844MB 敏感 CISA 数据，包括 AWS GovCloud 管理员密钥和数十个内部系统的明文密码——时间长达近 6 个月。

### 事件时间线

- **2025 年某时**：承包商创建了名为 "Private CISA" 的公开 GitHub 仓库，包含 `importantAWStokens`（AWS GovCloud 管理凭证）和 `AWS-Workspace-Firefox-Passwords.csv`（明文用户名密码）等文件
- **期间**：GitGuardian 的自动扫描系统**发出了 9 条告警邮件**——全部被忽视
- **2026 年 5 月 15 日**：GitGuardian 研究员 Guillaume Valadon 联系 KrebsOnSecurity 求助通知 CISA
- **通知后**：CISA 超过 48 小时才完成密钥轮换

### CISA 报告的关键教训

1. **密钥轮换太慢**：CISA 将延迟归因于与联邦和行业合作伙伴的系统互联复杂性
2. **外部报告渠道混乱**：研究者不得不尝试多种途径——邮件承包商、通过 CISA 漏洞披露平台（本意是面向社区漏洞）、最终找了记者
3. **事故预案遗漏云场景**：CISA 有事故响应手册，但"不知道怎么处理 GitHub 或其他云服务"

### GitGuardian 研究员的评价

> "让 9 封通知邮件无人回应，这就是一天的 incident 变成六个月 exposure 的方式。"

> "让报告泄露变得简单——不只是关于你的产品。向你报告泄露的人不是威胁。"

> "据我所知，这是第一次有一个国家网络安全机构公开倡导秘密扫描和简化与安全研究者的关系。这是我们应该期望每个组织都做到的事件沟通。"

CISA 最终给了自己及格分——增强的日志能力和零信任原则的采用帮助他们确认没有客户或任务数据被暴露。但这份报告的核心价值在于：**连美国网安机构都在最基本的密钥管理上翻了车**。

🔗 https://krebsonsecurity.com/2026/07/lessons-learned-from-cisas-recent-github-leak/

## 🛡️ 为什么不把整个栈都做成 guard page？Raymond Chen 的答案

**来源**：The Old New Thing（Raymond Chen）· 2026-07-13 14:00

Microsoft 资深工程师 Raymond Chen（在 Windows 团队工作超过 30 年）回答了一个看似合理的问题："既然 guard page 可以自动扩展栈，为什么不干脆把整个栈都做成 guard page？"

### 答案：一次 page fault 会变成灾难

如果整个栈都是 guard page，那么**一个远低于栈限制的 page fault 可能导致系统分配高达 1GB 内存**（如果程序默认栈大小设为 1GB）。在调试器里，你会看到一个内存读取操作耗时数分钟。

更糟糕的是，这发生在内核态——你在任务管理器里杀进程，**它不会死，只会继续膨胀**。

> "即使 guard page 超过一页，它仍然是一个小的固定数量——系统可以在很短时间内满足 guard page fault。更重要的是，工作量是有界的。"

简洁、经典 Raymond Chen：一个好的系统设计不只是"能不能工作"，而是"最坏情况下会怎样"。

🔗 https://devblogs.microsoft.com/oldnewthing/20260713-00/?p=112528
